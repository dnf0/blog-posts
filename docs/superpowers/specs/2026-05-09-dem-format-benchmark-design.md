# DEM Format Benchmark — Design Spec

**Goal:** Benchmark reading DEM data from COG at 30m resolution vs. the same data packaged as Parquet (with S2 and H3 indexing) and GeoParquet, comparing DuckDB, Rasterio, rioxarray, Polars, and pandas. Target hardware: M1 Mac with 8GB RAM. Output: a polished blog post with plots and tables for dnf0.github.io.

## Core thesis

You don't need a server for large geospatial data — if you pick the right format and tool for your access pattern. This benchmark demonstrates which combinations survive (and which die) on constrained hardware.

## Data preparation

- **Source:** Copernicus GLO-30 DEM at 30m resolution
- **Extent:** ~6°×6° (~22,000 px per side, ~1.8 GB uncompressed) — large enough to exceed RAM when naively loaded
- **Pipeline** (`scripts/data_prep.py`):
  1. Download COG tiles covering the extent, build VRT mosaic if needed
  2. Verify valid COG with internal overviews; convert with `gdal_translate -of COG` if not
  3. Extract row-per-pixel to flat Parquet (`x, y, band_value`)
  4. Add S2 cell column (level 14, ~0.03 km² per cell) → `dem_s2.parquet`
  5. Add H3 cell column (resolution 12, ~0.03 km² mean area) → `dem_h3.parquet`
  6. Add WKB geometry column (POINT) + write GeoParquet via `geopandas.to_parquet()` → `dem.geoparquet`
- Record on-disk file sizes for each format

## Benchmark harness

### Query types

| Query | Description |
|-------|-------------|
| Point sample | Elevation at 100 random lat/lon coordinates |
| Small bbox | 0.01°×0.01° window (~450×450 px) |
| Small polygon | Irregular polygon ~0.02° across |

### Metrics per run

Each row in `results/benchmarks.parquet`:
- `format`, `tool`, `query_type` — identifiers
- `duration_ms` — wall clock time for the query only
- `peak_memory_mb` — peak heap allocation from memray (total bytes allocated at peak)
- `filesize_mb` — on-disk size of the format file
- `memray_report_path` — path to the per-run memray binary capture for later inspection
- `success` — boolean (false = OOM, timeout, or crash)

### Memory profiling with memray

Each benchmark run is wrapped in a `memray.Tracker()` context:

```python
import memray

with memray.Tracker(f"results/memray/{format}_{tool}_{query_type}_run{i}.bin"):
    result = run_query(...)
```

Benefits over RSS sampling:
- **Actual heap allocations**, not virtual/RSS noise from mmapped files or shared libraries
- **Per-run flame graphs** can be generated afterward for blog visuals
- **Allocation timeline** shows whether memory spikes are transient or sustained
- **C API calls tracked** — shows what Rasterio/GDAL does under the hood vs. pure Python in Polars/DuckDB

Post-benchmark, `visualize.py` generates a **representative memray flame graph** for one run of each format×tool winner, showing where memory actually goes.

### Execution
- 10 runs per combination to capture variance
- `gc.collect()` and module cache clearing between runs
- Dataset handles closed between runs (prevents mmap bleed)
- 120-second timeout per individual benchmark
- memray binary captures saved to `results/memray/` (gitignored, large files)

### COG-specific tool notes
- **Rasterio:** Open dataset once per run, use `.read(window=)` for bbox queries, `.sample()` for points
- **rioxarray:** `open_rasterio(chunks=...)` with lazy loading, `.sel()` for spatial subset
- **DuckDB:** `LOAD spatial; SELECT * FROM ST_Read('dem.cog') WHERE ...`

### Parquet-specific tool notes
- S2/H3 query strategy: pre-compute which cell IDs cover the AOI, then `WHERE s2_cell IN (...)` or `WHERE h3_cell IN (...)`. The query engine just does integer filtering — spatial smarts are baked into the index column at conversion time.
- GeoParquet: DuckDB uses `ST_Within(geom, ST_GeomFromText('POLYGON(...)'))`, GeoPandas uses `.cx[]` or spatial join.

## Edge cases

- Missing COG tiles at region boundary → VRT handles partial coverage; Parquet extraction skips nodata
- Tool not installed → `benchmark.py` checks imports at startup, skips gracefully
- GeoParquet without DuckDB spatial extension → skip with note
- OOM or crash during benchmark → exception caught, `success=false`, memray report up to that point preserved
- Timeout → 120s per-benchmark cap via `multiprocessing.Process` with `.join(timeout)`, marked as timeout
- All runs for a combo fail → still included in results with `success=false` across the board

## Reproducibility

`config.py` pins: random seed, region extent as GeoJSON bbox, query points/polygons. Re-run against any region by changing `config.py` only.
