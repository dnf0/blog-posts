# Enforcing Engineering Rigour in AI Agents

*May 14, 2026 • 10 min read • Tags: `ai-workflows`, `orchestration`, `developer-experience`*

<Callout type="info">
**TL;DR:** Deploying raw AI models across an organisation leads to fragmented workflows and chaotic codebases. By adopting the unified `agent-rules` ecosystem—which seamlessly integrates cross-provider standardisation, stateful orchestration, and expert capabilities like `graphify` and `roborev-integration`—I was able to standardise developer experience without bloat and provide a foolproof onboarding path for non-engineers.
</Callout>

I was recently tasked at my current organisation with identifying a curated set of tooling to standardise our developer experience (DevEx) across AI-assisted workflows. The mandate was twofold: enforce engineering rigour without introducing heavy bureaucratic bloat, and provide a frictionless onboarding experience for non-engineering users venturing into terminal-based AI tooling like the Gemini CLI and Claude Code for the first time. 

When I initially observed unguided AI agents operating within our codebases, they acted without context, discipline, or standard operating procedures. The results were wildly unpredictable. An agent might write a brilliant algorithm one moment and execute a destructive, unverified refactor the next. Unconstrained AI is an organisational liability, not an asset. To fix this, I introduced strict workflow orchestration—forcing these models to abandon their erratic autonomy and adopt systematic engineering rigour.

## The Problem: Fragmented Ecosystems and Stateless Agents

My first observation was the severe fragmentation of the AI tooling ecosystem. Different teams were using a disparate array of IDEs and CLI tools—Cursor, Claude Code, and the Gemini CLI. Each tool required its own configuration schema, leading to massive context drift. An architectural rule established in `.cursorrules` was completely ignored by Claude. 

Furthermore, I discovered a critical limitation in relying purely on system prompts: static rules are inherently stateless. I could write a mandate to "always write tests," but in the heat of a complex implementation, an agent would routinely skip verification if it hit a roadblock. I needed a mechanism to force the AI to empirically validate its assumptions before claiming a task was complete.

## Curating the Lean Stack

To solve this, I deliberately avoided heavy, monolithic platforms. Instead, I curated a subset of highly composable tools that, in combination, provided everything needed for a rigorous engineering organisation. Standard tools operate optimistically: unconstrained agents will plan, execute, and rely on their own internal logic to assume success, often bypassing rigorous empirical validation. To combat this, I constructed a closed-loop engineering cycle grounded in classical software engineering principles—specifically, the triad of **Understanding**, **Execution**, and **Validation**.

### Unification via agent-rules

Before establishing the loop, I had to resolve the fragmentation problem. I deployed the `agent-rules` repository to act as the single source of truth. I authored our core engineering mandates in a universal `AGENTS.md` file. The tooling then dynamically compiled this master file into native formats tailored for each provider (e.g., `.cursorrules`, `CLAUDE.md`, `GEMINI.md`). 

### Understanding: "Measure Twice, Cut Once" via Graphify

*Engineering Principle: You cannot safely modify a complex system you do not fully understand.*

Standard agents operate in a vacuum, relying on keyword searches or limited context windows, which frequently results in destructive "blind refactoring." To solve the "Understanding" phase, I integrated `graphify` [2](#references). Before any structural changes are planned or executed, Graphify forces the agent to map the architectural dependencies of the entire codebase into a clustered knowledge graph. This ensures the AI has a mathematically verified understanding of the system's boundaries and dependencies *before* it writes a single line of code.

### Execution: The "Red-Green-Refactor" Loop via Superpowers

*Engineering Principle: Test-Driven Development (TDD) prevents regressions and forces empirical proof of correctness.*

Establishing static rules was insufficient; I needed to orchestrate the actual workflow. I integrated `@obra/superpowers` [1](#references) to act as the execution orchestration layer. Standard AI tools are inherently brittle—if they encounter a failing test, they often bypass verification and blindly claim victory. Superpowers overrides these default behaviours by forcing agents to follow an inescapable, stateful state machine. It mandates strict TDD cycles. If a test fails, the agent cannot simply move on; the orchestrator routes it back into a systematic debugging loop until empirical validation is achieved.

### Validation: "Trust, but Verify" via Roborev

*Engineering Principle: Author-driven testing is insufficient without independent, adversarial peer review.*

Finally, the loop required independent validation. Standard agents reviewing their own work is a fundamentally flawed paradigm—they suffer from confirmation bias and hallucinated logic. To close the feedback loop, I integrated `roborev` [3](#references). Following the execution phase, Roborev locks the agent into a strict, adversarial code review loop. It forces the agent to independently defend its implementation against the original specification. If the logic is flawed or the tests are superficial, Roborev rejects the commit, forcing the agent back to the Superpowers debugging loop. This ensures every line of code is rigorously proven before a human developer ever reviews a pull request.

## Zero-to-Hero: Onboarding Non-Engineers

Architectural purity is functionally useless if it creates friction for adoption. My second mandate was to ensure that non-engineering users—product managers, technical writers, and analysts—could leverage these tools without facing a steep technical learning curve.

I engineered "zero-to-hero" bootstrap scripts and prebuilt templates within the `agent-rules` ecosystem. With a single shell command, any user could inject this entire cognitive architecture into their local environment. 

This democratisation was the final piece of the puzzle. Non-engineers did not need to understand the intricacies of stateful orchestration, cross-provider generation, or TDD. They simply ran the bootstrap script and immediately inherited a disciplined, highly capable AI assistant that was pre-configured to adhere to the organisation's rigorous standards.

## Conclusion: A Cohesive DevEx

By deploying the unified `agent-rules` ecosystem—which intrinsically bundles stateful orchestration and expert skills directly into its standardisation pipeline—I was able to standardise the developer experience across our entire organisation without introducing unnecessary bloat. AI models are only as effective as the workflows that constrain them; true productivity in the AI era requires replacing erratic autonomy with systematic orchestration.
