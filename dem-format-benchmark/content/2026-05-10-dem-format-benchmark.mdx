# Effective Querying of Global Multi-Dimensional Data Sets

*May 10, 2026 • Tags: `geospatial`, `python`, `rust`, `zarr`, `polars`, `benchmarking`*

<Callout type="info">
**TL;DR:** For batch extraction of 1K+ polygons, a pure-Rust backend querying sharded N-dimensional Zarr arrays provides an optimal balance of storage compression, analytical accuracy, and synchronous API performance.
</Callout>

The production requirement is to build a high-concurrency web API serving global climate hazard data. The spatial resolution is 30m across all of Earth's landmass, equating to approximately 166 billion pixels per layer.

When accounting for the dimensionality of climate modeling—12 years, 4 emissions scenarios, 5 return periods, and 3 percentiles—the dataset expands to 720 distinct layers per spatial pixel. This results in a multi-dimensional data cube containing over 120 trillion data points and exceeding 240 Terabytes in uncompressed size.

A primary constraint of this system is that the API must execute synchronous, low-latency extractions for batches of up to 1,000 irregular polygons at a time. If a user queries 1,000 catchments or property boundaries on a map, the hazard analytics must be returned within synchronous API timeout limits.

To empirically test architectures for this challenge, we use a 6°×6° Copernicus Digital Elevation Model (DEM) [1] covering the Swiss Alps (1.7GB, ~466M pixels). By applying a `q2500` quantization (rounding elevations into 2,500m buckets), we artificially transform the varying mountain terrain into uniform blocks (see Figure 1). 

![Quantization Comparison](/plots/00_quantization_visual.png)
*Figure 1: Visual comparison of raw elevation data vs. `q2500` quantized terrain, simulating the smooth nature of categorical hazard models.*

This provides a reasonable approximation of the categorical nature of global hazard data (such as flood models or climate bands), allowing us to rigorously benchmark how different formats handle spatial scaling.

## Exploring the Ecosystem

The standard approach to managing spatial raster data involves N-dimensional arrays, predominantly Cloud Optimized GeoTIFFs (COG) [2] or cloud-native Zarr stores [3]. 

In these formats, data is chunked into 2D or 3D grids (e.g., 512x512 pixels) and compressed using algorithms like ZSTD or DEFLATE. They are highly efficient at storage because the X and Y coordinates are implied by the pixel's position in the grid array. The highly quantized DEM proxy compressed to 2.4 MB in COG and 1.7 MB in Zarr.

However, their query engines are generally not optimized for large-batch polygon extractions. When querying 1,000 irregular polygons using standard ecosystem tools (such as `rasterio.mask` for COGs or `rioxarray.clip` for Zarr), the system performs sequential raster masking. 

The underlying process typically involves:
1. Calculating the bounding box for a polygon.
2. Fetching and decompressing the intersecting spatial chunks.
3. Rasterizing the polygon boundary into a binary mask array in memory.
4. Multiplying the mask against the data array to extract the values.

This constitutes an $O(N)$ sequential operation over geometries. 

Extracting 1,000 polygons via Rasterio took 0.73 seconds. This benchmark represents a best-case scenario: it assumes the entire DEM is stored in a single, locally mounted TIFF file. In production, global raster datasets are stored as numerous partitioned map tiles in cloud object storage (e.g., AWS S3). Querying 1,000 global polygons against tiled COGs would require substantial network I/O, downloading numerous remote HTTP headers, and dynamically stitching tiles in memory, leading to significantly higher latency. 

The standard Zarr Python ecosystem (`xarray` and `rioxarray`) encountered severe performance limitations, requiring over 13 seconds to process 1,000 polygons due to the overhead of instantiating Python objects and building dependency graphs for each individual extraction.

This linear scaling is insufficient for synchronous Web API constraints, necessitating an alternative paradigm.

## Tabular Joins and the Coordinate Tax

![Storage size by format](/plots/01_storage_size.png)
*Figure 2: Storage footprint of different architectural formats, illustrating the impact of the Coordinate Tax and Quadtree Compaction.*

An alternative approach is treating spatial data as tabular data (see Figure 2). By flattening the data into a columnar format like Parquet [4], we can utilize relational database engines like DuckDB [5] or Polars [6].

A known limitation of this approach is the "Coordinate Tax". Traditional Parquet and GeoParquet store explicit `x` and `y` coordinates (or WKB geometries) for every pixel. This introduces significant storage overhead. A flat Parquet file for the proxy DEM required hundreds of megabytes.

To eliminate this overhead, we utilized a Native 2D Hilbert Curve [7]. By mapping the global 1-arc-second grid `(col, row)` to a 1D Hilbert integer using bitwise XOR rotations, the index replaces the explicit location coordinates.

### Quadtree Compaction

Because the hazard data proxy contains contiguous uniform regions, compression can be improved through Quadtree Compaction. 

Unlike COG and Zarr (which compress fixed blocks using DEFLATE/ZSTD), Quadtree Compaction recursively merges identical spatial data using bit-shifting. The hierarchy is encoded into the integer itself: shifting a Hilbert code to the right by 2 bits (`d >> 2`) yields the ID of its parent. If four identical adjacent 30m child pixels share the same elevation value, they are replaced by a single 60m parent cell. 

This process recursively ascends 16 levels of the quadtree. For uniform areas, millions of pixels collapse into larger parent cells, functioning as a 2D Run-Length Encoder. 

Applying this to the Hilbert Parquet file reduced the file size from 1.7 GB to 2.8 MB. (Note that this is slightly larger than COG at 2.37 MB and Zarr at 1.65 MB, because Parquet must still explicitly store the 1D Hilbert index column for every cell, whereas N-dimensional arrays imply locations entirely by position).

### Evaluating Vectorized Hash Joins

With the data stored as compacted 1D integers in Parquet, we implemented a client-side region coverer. The system calculates the exact Hilbert cells intersecting the requested polygons and passes that list to Polars or DuckDB to execute a vectorized Hash Join.

![Polygon Scaling](/plots/05_scaling_polygons.png)
*Figure 3: Scaling execution time as a function of polygon batch size. Database join logic scales sub-linearly, while traditional bounding box masks scale linearly.*

As seen in the scaling chart (see Figure 3), the database query step (the Hash Join) scales sub-linearly (empirically $O(N^{0.83})$). Polars processed the database query for 1,000 polygons in 0.18 seconds. 

This is primarily because 466 million raw DEM pixels were compacted down to roughly 2.6 million parent cells, mathematically shrinking the problem space before the join operation.

## The Area-Weighting Trade-off

However, this compaction introduces a mathematical side-effect relevant for production analytics: the loss of area-weighting.

When executing an `AVG(band_value)` query, SQL treats each row equally. In a compacted Quadtree, one row might represent a single pixel, while another represents a uniform region of 1,024 pixels. 

If a polygon covers both regions, the database returns two rows. The single pixel will disproportionately affect the average elevation because the 1,024 pixels were compacted into a single row. Furthermore, a `pixel_count` workaround fails on polygon boundaries because it over-weights partially touched cells.

**The Boundary Paradox in Practice:** We conducted an empirical equivalence test comparing Rasterio's C++ GDAL heuristics with a pure-Rust geometric masking approach. For blocky polygons, the outputs were identical (Median Absolute Error of 0.0000 meters). However, for linear polygons with high perimeter-to-area ratios (such as a 10m buffered pipeline), floating-point boundary tie-breakers become significant. GDAL's Center-Point heuristics might include 180 pixels, while an "All-Touched" equivalent includes 540 pixels—a 200% discrepancy that alters the resulting average.

**Structuring Parquet for Analytics:** The optimal Parquet structure depends on the SQL aggregation used:
1. **For `MAX()` and `MIN()`:** Quadtree Compaction is mathematically sound. Since a cell is only compacted if all child pixels share the exact same value, intersection guarantees the correct value. The 2.8 MB compacted file is highly efficient for extreme-value queries.
2. **For `AVG()` and `SUM()`:** The `pixel_count` workaround fails on boundary cells, making Uncompacted Parquet the mathematically correct option. Skipping the Quadtree Compaction step ensures every pixel retains its own row. Thanks to Parquet's native ZSTD/RLE compression, an uncompacted `q2500` Hilbert file still compressed to 319 MB, retaining pixel-level analytical accuracy while supporting efficient Hash Joins.

## Native Zarr Integration in Rust

While the Parquet Hash Join is efficient, generating the Hilbert pixel masks for 1,000 polygons (the "Covering" step) took approximately 0.27 seconds, creating a preprocessing bottleneck. 

To achieve faster geometric extraction while maintaining raw pixel mathematical accuracy and chunked array compression, we evaluated a native Zarr approach.

Bypassing the Python object overhead, we built a custom PyO3 plugin using the pure-Rust `zarrs` crate [8], combined with `geo-rasterize` [9] and `rayon` [10] for multi-threading.

**Performance:** The Rust binary reads the `.zarray` metadata directly, traces the pixel bounds for every polygon using the pure-Rust rasterizer, fetches the necessary bytes from the chunked Zarr array, and tallies the result natively. 

![Polygon Extraction Latency](/plots/04_polygon_latency.png)
*Figure 4: Median extraction latency for 1,000 polygons across evaluated architectures. (Note: Hatched bars indicate architectures that timed out and failed to complete within the strict 30-second synchronous API constraint).*

This pipeline extracted 1,000 polygons in 0.24 seconds (see Figure 4). It is mathematically a linear $O(N)$ operation. However, because the geometric masking was shifted into native Rust structs operating in the CPU's L1 cache, the constant factor is sufficiently small that the scaling curve remains low relative to traditional methods.

**Cloud Latency Considerations:** This 0.24-second execution was run locally. Prior to each benchmark run, the OS page cache was explicitly cleared to ensure a "cold start." However, within a massive batch execution, significant intra-run caching occurs. Because the 6°×6° proxy DEM generates only ~1,700 Zarr chunks, querying thousands of localized polygons mathematically guarantees that the same chunks are accessed repeatedly. The first few thousand polygons pull the entire dataset into RAM, allowing subsequent polygons to read from a "hot cache." In a true 240 TB global deployment, querying thousands of disparate polygons globally would trigger constant cache misses, forcing continuous network I/O. While S3 performance was not empirically benchmarked, Zarr's architecture is positioned to mitigate this cloud latency. Tiled COGs require downloading TIFF headers for multiple disparate files, whereas Zarr uses a single JSON metadata file. Because Native Zarr fetches raw byte-ranges, an asynchronous Rust backend could execute concurrent HTTP Range requests for the exact chunks needed.

Because it operates natively on the raw pixels rather than compacted cells, this approach is immune to the Parquet area-weighting flaw, yielding correct `AVG()` aggregations natively.

### Comparing Zarr and Parquet Compression
Zarr's file size (1.7 MB) is actually smaller than the compacted Parquet file (2.8 MB). This is because Zarr requires a dense, regular grid; it stores all 466 million pixels and relies on algorithms like ZSTD or DEFLATE to compress them, which is incredibly efficient when locations are implied by position. Parquet requires 2.8 MB because, even though Quadtree Compaction explicitly removed millions of identical rows, it still must explicitly store the 1D Hilbert index column for every remaining parent cell. 

Furthermore, Zarr's default compression traverses chunks in standard row-major (C-contiguous) order, which breaks spatial locality at the end of every line and reduces ZSTD efficiency. Reordering the 2D chunk bytes along a Hilbert curve via a custom `numcodecs` filter prior to compression would maximize the run-length of identical values, potentially improving Zarr's compression ratio to match Parquet without altering the dense array structure.

## Spatial Chunking vs. Row-Level Random Access

While Zarr v3 and Parquet are established formats, emerging technologies like Lance [12] are actively addressing spatial data limitations. Lance is a modern columnar format written in Rust specifically designed for machine learning and multimodal data. It natively integrates with GeoArrow. Lance is optimized for huge-scale data storage and rapid random access. In recent benchmarks, Lance demonstrated significantly faster random access times than Parquet for fetching specific blobs of data (like imagery chunks). If a spatial API requires feeding randomly shuffled data into an AI model for inference, Lance provides an optimized solution.

The distinction between Lance and Zarr for polygon queries relies on the difference between Spatial Locality and Row-Level Retrieval.

When querying a polygon, the requirement is a clustered geographic neighborhood. Zarr is optimized for this via 2D spatial tiles (e.g., 512x512 pixels). The Rust backend fetches the required 2D tile via a single HTTP Range request. Once decompressed in memory, all necessary pixels are spatially contiguous.

Lance and Parquet are 1D columnar formats. Querying a polygon requires translating the 2D shape into thousands of 1D Hilbert integers. Because the irregular polygon boundary cuts across the Hilbert curve, these integers are fragmented, requiring Lance to execute thousands of non-contiguous row fetches. The overhead of assembling these fragmented rows limits its performance for massive batch polygon extractions (~0.46 seconds for 1K polygons, compared to Zarr's 0.24 seconds).

#### The Read Amplification Paradox (Overfetching)
A common critique of Zarr is that it suffers from "overfetching" (Read Amplification). If a polygon only touches 10 pixels within a 512x512 chunk, Zarr still must download and decompress the entire 262,144-pixel chunk. 

While this is true, it is a paradox that Parquet and Lance actually overfetch *more* data for 2D geometries. Columnar formats compress data into discrete blocks (e.g., 1MB Parquet Pages). To read a single row, the database must download and decompress the entire 1MB Page. Because a 2D polygon fragments into thousands of disconnected 1D Hilbert segments, those segments scatter across dozens of different compression pages on disk. 

To empirically prove this, we tracked the exact disk I/O required to extract a single polygon requiring just **23 KB** of raw pixel data (see Figure 5):

![Overfetch Comparison](/plots/06_overfetch_comparison.png)
*Figure 5: Log-scaled comparison of the uncompressed data fetched by Zarr (at various 2D chunk sizes) versus Parquet (1D Row Groups) to extract a 23 KB polygon.*

*   **Zarr (512x512 chunks):** Touched 1 chunk, downloading **0.50 MB** of uncompressed data.
*   **Parquet (Row Groups):** The 1D Hilbert line wandered so widely that the polygon's min/max bounds spanned 735 different Parquet Row Groups. DuckDB was forced to download and decompress **689 MB** of irrelevant data just to find the 23 KB it needed!

**The Parquet Zone Map Trap:** Why did DuckDB read 689 MB of data? Parquet uses "Zone Maps" (min/max metadata per Row Group) to quickly skip irrelevant data (Predicate Pushdown). When DuckDB queries the polygon, it looks at the absolute lowest and highest Hilbert integer inside the polygon and tells Parquet to fetch every Row Group that falls within that range. Because the 1D Hilbert curve snakes back and forth across the 2D map, a tiny geographic polygon can span a massive numeric range. DuckDB fetched 735 Row Groups, decompressed them into RAM, and then realized 99% of the rows belonged to neighboring towns outside the polygon. 

**The Coordinate Tax Reconsidered (2D Zone Maps):** Is there a way to solve this Parquet overfetching without abandoning the format? Yes, but it requires paying the "Coordinate Tax" we initially tried to avoid. If we add the explicit X and Y coordinates (or a coarse 2D `grid_id`) back into the uncompacted Parquet file, the file size bloats from 319 MB to roughly 528 MB. However, because Parquet maintains Zone Maps for *every* column, we can structure our query to include a 2D bounding box: `WHERE x >= min_x AND x <= max_x AND z_index IN (...)`. DuckDB will evaluate the X/Y Zone Maps first, allowing it to instantly prune the hundreds of irrelevant Row Groups that the 1D Hilbert index dragged in. We empirically tested this, and DuckDB's scan time dropped to near zero. This represents the ultimate architectural tradeoff: you can pay for storage (528 MB Parquet) to fix your I/O, or you can use Native Zarr to keep the file size tiny (9.8 MB) while inherently maintaining 2D chunk bounds.

Therefore, to read a single polygon, a columnar database relying solely on a 1D spatial index downloads and decompresses significantly more irrelevant data blocks than Zarr, which cleanly bounds the overfetching to a few 2D spatial chunks.

**Tuning the Spatial Chunk Size (Read vs. Request Amplification):** It is important to note that the 512x512 chunk size is simply a default designed to balance compression ratios with read speeds. If a production API primarily queries very small polygons, fetching a 0.50 MB chunk is still an overfetch. The logical solution is to configure the Zarr writer to use smaller chunks. In our simulation for the exact same 23 KB polygon, configuring Zarr with `chunks=(128, 128)` caused it to touch 4 chunks, reducing the total download size to just **0.12 MB**. 

However, doing so introduces the fundamental trade-off of cloud I/O: **Read Amplification vs. Request Amplification**. 
While smaller chunks drastically reduce the number of wasted bytes downloaded, they increase the number of HTTP Range requests required to cover a polygon (e.g., 4 requests instead of 1). In cloud storage like S3, Time-To-First-Byte (TTFB) latency is exceptionally high. Making 100 small HTTP requests is almost always slower and more expensive than making 1 large HTTP request. 

Data engineers must tune the chunk size to find the exact sweet spot for their specific API payloads. Conversely, while Parquet's block sizes (Data Pages and Row Groups) can also be shrunk to reduce 1D overfetching, doing so breaks the contiguous lengths required for its columnar Run-Length Encoding (RLE) and dictionary compression, causing the file size to explode. Zarr's 2D chunks avoid this tension, allowing for true geometric tuning.

**General Heuristic:** For fetching disconnected data points (e.g., AI model data shuffling), Lance is optimal. For fetching clustered 2D shapes, Zarr is highly efficient.

## Scaling Back to the Globe

Returning to the original 240 TB multi-dimensional problem (720 distinct hazard layers per 30m spatial pixel), the architectural path requires evaluating the limits of modern data structures. 

Scaling the Hilbert Parquet architecture globally introduces a hyper-dimensionality challenge. Parquet schemas are designed for narrow tables. Storing 720 layers per pixel requires either an exceptionally wide table (destroying columnar scan efficiency) or a deeply nested array structure (forcing the database to deserialize large blobs of data to read a single variable). 

Conversely, Zarr natively handles N-dimensional chunking (e.g., `[layer, y, x]`). If an API request specifies a specific layer, the Rust backend calculates the 1D index for that slice and reads only the corresponding bytes, leaving the other layers untouched.

### Overcoming Cloud Storage Limits: Zarr v3 Sharding
A final consideration for deploying a global Zarr cube is the nature of cloud object storage. 

Zarr v2 stores every chunk as an individual file. A global 30m dataset with 720 layers chunked into 512x512 grids would generate billions of files. In a key-value object store like S3, this volume of files exhausts inode limits on caching layers, slows data transfer, and increases costs associated with HTTP `PUT` and `LIST` requests. 

The standard solution is Zarr v3 Sharding [11]. Sharding introduces a two-tier hierarchy by grouping thousands of chunks into larger "Shard" files (e.g., 500 MB). A byte-offset index is appended to the end of each shard. 

When the Rust backend queries the globe, it:
1. Reads the master metadata to locate the relevant Shard file.
2. Fetches the trailing byte-offset index via an HTTP Range request.
3. Parses the index to locate the byte-offset of the specific 512x512 inner chunk.
4. Executes a second HTTP Range request to fetch the required bytes.

This approach preserves the read performance of small chunks while drastically reducing the total number of files in the storage bucket. Furthermore, because global hazard data typically follows a "Write Once, Read Many" (WORM) lifecycle, the shard index never changes. The Rust backend can aggressively cache these indices in RAM. This entirely neutralizes the "double-fetch" penalty of Sharding, allowing all subsequent API queries to instantly locate the inner chunk and fetch the required data with a single HTTP Range request.

## Conclusion
For large-scale, multi-dimensional spatial analytics, traditional GIS raster pipelines face constraints due to sequential $O(N)$ geometry masking. Tabular Parquet hash joins offer a sub-linear alternative for 1D datasets, but present challenges when calculating exact area-weighted averages or scaling to highly dimensional schemas. 

For a high-concurrency, global hazard API, a pure-Rust backend querying sharded N-dimensional Zarr v3 arrays provides a robust balance of storage compression, analytical accuracy, and synchronous API performance.
