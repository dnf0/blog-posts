# Effective Querying of Global Multi-Dimensional Data Sets

*May 10, 2026 • 18 min read • Tags: `geospatial`, `python`, `rust`, `zarr`, `polars`, `benchmarking`, `data-engineering`*

<Callout type="info">
**TL;DR:** To build a high-concurrency API capable of synchronously querying hundreds of terabytes of multi-dimensional global hazard data, you must overcome the $O(N)$ bottlenecks of traditional raster masking. After benchmarking several architectures, I found that for batch extraction of 1K+ exclusive (i.e., non-overlapping) globally disparate polygons representing distinct queries—with the ideal goal of loading many thousands per second—a pure-Rust backend querying sharded N-dimensional Zarr arrays provides an optimal balance of storage compression, analytical accuracy, and synchronous API performance.
</Callout>

The overarching objective of this engineering initiative is to design and build a high-concurrency web API capable of serving global climate hazard data at scale. The physical scope of this task is substantial: the target spatial resolution is 30m across all of Earth's landmass, which equates to approximately 166 billion pixels per layer. 

This scale compounds rapidly when factoring in the dimensionality required by modern climate modelling. For instance, in my current role, I am working with data that spans 12 projection years across multiple emission scenarios, 5 return periods, and 3 percentiles. This expands the dataset to over 700 distinct layers per spatial pixel, resulting in a multi-dimensional data cube containing over 120 trillion data points and exceeding 240 Terabytes in its uncompressed state.

A critical constraint of this API is the requirement for synchronous, low-latency execution. Specifically I am working on a system which must process batch extractions of thousands of irregular polygons simultaneously. If an end-user queries 1,000 distinct catchments or property boundaries on a map, the hazard analytics for all 1,000 polygons must be calculated and returned within standard synchronous API timeout limits.

To empirically test and benchmark architectures capable of meeting these strict constraints, I use a 6°×6° Copernicus Digital Elevation Model (DEM) [1] with a matching 30-meter spatial resolution (GLO-30) covering the Swiss Alps (1.7GB, ~466M pixels) as my proxy dataset. Because global hazard data (such as flood inundation models or climate classification bands) is inherently categorical, I artificially simulate this data profile by applying multiple levels of quantization (`q100`, `q1000`, and `q2500`). These quantization passes round the continuous elevation values into distinct buckets, transforming the highly variable mountain terrain into uniform spatial blocks (see Figure 1). I evaluate multiple quantizations to artificially control the spatial contiguity of the data. A fine `q100` quantization produces highly volatile, fragmented regions, while a coarse `q2500` quantization produces substantial, smooth, uniform blocks. This allows me to rigorously benchmark how different compression algorithms (like [Quadtree Compaction](https://en.wikipedia.org/wiki/Quadtree) and [Run-Length Encoding](https://en.wikipedia.org/wiki/Run-length_encoding)) scale when encountering varying levels of physical spatial contiguity.

## Benchmark Environment & Hardware Constraints
To ensure reproducibility and transparency, it is critical to define the constraints of the benchmark environment:
*   **Hardware:** Apple MacBook M4, 12 cores, 24 GB RAM.
*   **Storage:** Local NVMe SSD.
*   **Concurrency Model:** To ensure a fair architectural comparison, I explicitly bypassed the Python Global Interpreter Lock (GIL) wherever possible. For the Rasterio benchmark, I used Python's `ThreadPoolExecutor` because Rasterio's underlying C++ bindings release the GIL during masking operations, allowing it to use all 12 cores. The pure-Rust backend (`zarrs` and `geo-rasterize`) and the Hilbert coverer (`polars_hilbert`) explicitly used multi-threading via the `rayon` crate. DuckDB and Polars naturally used multi-threading for their tabular Hash Joins. Only the Python-native `xarray` operations remained bound by single-threaded execution loops. The fact that the 12-core pure-Rust Zarr implementation significantly outperformed the 12-core Python/C++ Rasterio implementation highlights the substantial overhead of Python object instantiation and serialization, even when the GIL is bypassed. 

## Methodology: Evaluated Architectures
To fully understand the results, it is essential to detail the different architectural combinations evaluated in this benchmark. I tested distinct pairs of data formats and query engines to extract data for geometric polygons:

1. **Rasterio + COG:** The industry-standard Python bindings for the C++ GDAL [13] library querying Cloud Optimised GeoTIFFs (COGs) [2]. It extracts data by individually masking the bounding box of each polygon. 
2. **Xarray + Zarr:** A popular Python data science library [14] used to lazily load and sequentially query N-dimensional Zarr arrays [3]. 
3. **DuckDB + Parquet:** An embedded analytical SQL database written in C++ [5] executing vectorized "Hash Joins" over the 1D columnar Parquet format [4].
4. **Polars + Parquet:** A high-performance DataFrame library written in Rust [6], used here to perform the exact same Hash Joins over Parquet as DuckDB, serving as a secondary validation of the columnar tabular approach.
5. **Pure Rust Zarr (`zarrs` + Zarr):** A custom backend I built natively in Rust. It bypasses Python and C++ entirely, directly fetching byte-ranges from Zarr chunks using the `zarrs` crate [8] and geometrically masking them using the `geo-rasterize` crate [9].
6. **DuckDB + Lance & Lance Scanner:** I queried the Lance data format [12]—optimised for rapid random access of data blobs for artificial intelligence (AI) models—using both DuckDB Hash Joins and the native Lance Scanner query engine.

## Exploring the Ecosystem

The standard approach to managing spatial raster data involves N-dimensional arrays, predominantly Cloud Optimised GeoTIFFs (COG) [2] or cloud-native Zarr stores [3]. In these formats, data is chunked into 2D or 3D grids (e.g., 512x512 pixels) and compressed using algorithms like ZSTD or DEFLATE. They are highly efficient at storage because the X and Y coordinates are implied by the pixel's position in the grid array. The highly quantized DEM proxy compressed to 2.4 MB in COG and 1.7 MB in Zarr (see Figure 2). However, their query engines are generally not optimised for large-batch polygon extractions. When querying 1,000 irregular polygons using standard ecosystem tools (such as `rasterio.mask` for COGs or `rioxarray.clip` for Zarr), the system performs sequential raster masking. The underlying process typically involves:
1. Calculating the bounding box for a polygon.
2. Fetching and decompressing the intersecting spatial chunks.
3. Rasterizing the polygon boundary into a binary mask array in memory.
4. Multiplying the mask against the data array to extract the values.

This constitutes an $O(N)$ sequential operation over geometries. 

Extracting 1,000 polygons via Rasterio took 0.73 seconds (see Figure 4). This benchmark represents a best-case scenario: it assumes the entire DEM is stored in a single, locally mounted TIFF file. In production, global raster datasets are stored as numerous partitioned map tiles in cloud object storage (e.g., AWS S3). Querying 1,000 global polygons against tiled COGs would require substantial network I/O, downloading numerous remote HTTP headers, and dynamically stitching tiles in memory, leading to significantly higher latency. The standard Zarr Python ecosystem (`xarray` and `rioxarray`) demonstrated even more substantial performance limitations, requiring over 13 seconds to process 1,000 polygons due to the overhead of instantiating Python objects and building dependency graphs for each individual extraction. This linear scaling is insufficient for synchronous Web API constraints, necessitating an alternative paradigm.

## Tabular Joins and the Coordinate Tax

An alternative approach is treating spatial data as tabular data. By flattening the data into a columnar format like Parquet [4], you can use analytical SQL engines like DuckDB [5] or DataFrame libraries like Polars [6]. A known limitation of this approach is the "Coordinate Tax". Traditional Parquet and GeoParquet store explicit `x` and `y` coordinates (or WKB geometries) for every pixel. This introduces significant storage overhead. An uncompacted, flat Parquet file with raw explicit X/Y coordinates for the 6°×6° proxy DEM required over 1.1 GB. To eliminate this overhead, I used a Native 2D Hilbert Curve [7]. By mapping the global 1-arc-second grid `(col, row)` to a 1D Hilbert integer using bitwise XOR rotations, the index replaces the explicit location coordinates.

### Quadtree Compaction

Because the hazard data proxy contains contiguous uniform regions, compression can be improved through Quadtree Compaction. Unlike COG and Zarr (which compress fixed blocks using DEFLATE/ZSTD), Quadtree Compaction recursively merges identical spatial data using bit-shifting. The hierarchy is encoded into the integer itself: shifting a Hilbert code to the right by 2 bits (`d >> 2`) yields the ID of its parent. If four identical adjacent 30m child pixels share the same elevation value, they are replaced by a single 60m parent cell. This process recursively ascends 16 levels of the quadtree. For uniform areas, millions of pixels collapse into larger parent cells, functioning as a 2D Run-Length Encoder. Applying this Quadtree Compaction to the `q2500` Hilbert Parquet file reduced the file size from the 1.1 GB raw baseline down to just 2.8 MB. (Note that this compacted Parquet file is slightly larger than the `q2500` COG at 2.4 MB and Zarr at 1.7 MB, because Parquet must still explicitly store the 1D Hilbert index column for every cell, whereas N-dimensional arrays imply locations entirely by position). (see Figure 2).

## The Area-Weighting Trade-off

However, this compaction introduces a mathematical side-effect relevant for production analytics: the loss of area-weighting. When executing an `AVG(band_value)` query, SQL treats each row equally. In a compacted Quadtree, one row might represent a single pixel, while another represents a uniform region of 1,024 pixels. If a polygon covers both regions, the database returns two rows. The single pixel will disproportionately affect the average elevation because the 1,024 pixels were compacted into a single row. Furthermore, a `pixel_count` workaround fails on polygon boundaries because it over-weights partially touched cells.

**The Boundary Paradox in Practice:** I conducted an empirical equivalence test comparing Rasterio's C++ GDAL heuristics with a pure-Rust geometric masking approach. For blocky polygons, the outputs were identical (Median Absolute Error of 0.0000 metres). However, for linear polygons with high perimeter-to-area ratios (such as a 10m buffered pipeline), floating-point boundary tie-breakers become significant. For instance, GDAL's default "Centre-Point" heuristic only includes a pixel if the geometric centre of that pixel falls strictly inside the polygon boundary. For a thin pipeline, this strict rule might only capture 180 pixels. Conversely, an "All-Touched" heuristic includes the pixel if *any* part of its area intersects the polygon, which might capture 540 pixels for that exact same pipeline. This creates a 200% discrepancy in the number of pixels sampled, fundamentally altering the resulting average elevation.

**Structuring Parquet for Analytics:** The optimal Parquet structure depends on the SQL aggregation used:
1. **For `MAX()` and `MIN()`:** Quadtree Compaction is mathematically sound. Since a cell is only compacted if all child pixels share the exact same value, intersection guarantees the correct value. The 2.8 MB compacted file is highly efficient for extreme-value queries.
2. **For `AVG()` and `SUM()`:** The `pixel_count` workaround fails on boundary cells, making Uncompacted Parquet the mathematically correct option. Skipping the Quadtree Compaction step ensures every pixel retains its own row. Thanks to Parquet's native ZSTD/RLE compression, an uncompacted `q2500` Hilbert file still compressed to 319 MB, retaining pixel-level analytical accuracy while supporting efficient Hash Joins.

## Native Zarr Integration in Rust

While the Parquet Hash Join is efficient, generating the Hilbert pixel masks for 1,000 polygons (the "Covering" step) took approximately 0.27 seconds, creating a preprocessing bottleneck. To achieve faster geometric extraction while maintaining raw pixel mathematical accuracy and chunked array compression, I evaluated a native Zarr approach. Bypassing the Python object overhead, I built a custom PyO3 plugin using the pure-Rust `zarrs` crate [8], combined with `geo-rasterize` [9] and `rayon` [10] for multi-threading.

**Cloud Latency Considerations:** This 0.24-second execution was run locally. Prior to each benchmark run, the OS page cache was explicitly cleared to ensure a "cold start." Because the 6°×6° proxy DEM generates only ~1,700 Zarr chunks, I specifically capped the benchmark at 1,000 polygons. Querying larger batches (e.g., 100,000 polygons) on this localized dataset would mathematically guarantee that the same chunks are accessed repeatedly, transforming the test into an artificial "hot cache" in-memory benchmark. In a true 240 TB global deployment, querying thousands of disparate polygons globally would trigger constant cache misses, forcing continuous network I/O. While S3 performance was not empirically benchmarked, Zarr's architecture is positioned to mitigate this cloud latency. Tiled COGs require downloading TIFF headers for multiple disparate files, whereas Zarr uses a single JSON metadata file. Because Native Zarr fetches raw byte-ranges, an asynchronous Rust backend could execute concurrent HTTP Range requests for the exact chunks needed.

Because it operates natively on the raw pixels rather than compacted cells, this approach is immune to the Parquet area-weighting flaw, yielding correct `AVG()` aggregations natively.

### Comparing Zarr and Parquet Compression
Zarr's file size (1.7 MB) is actually smaller than the compacted Parquet file (2.8 MB) (see Figure 2). This is because Zarr requires a dense, regular grid; it stores all 466 million pixels and relies on algorithms like ZSTD or DEFLATE to compress them, which is incredibly efficient when locations are implied by position. Parquet requires 2.8 MB because, even though Quadtree Compaction explicitly removed millions of identical rows, it still must explicitly store the 1D Hilbert index column for every remaining parent cell. 

Furthermore, Zarr's default compression traverses chunks in standard row-major (C-contiguous) order, which breaks spatial locality at the end of every line and reduces ZSTD efficiency. Reordering the 2D chunk bytes along a Hilbert curve via a custom `numcodecs` filter prior to compression would maximise the run-length of identical values, potentially improving Zarr's compression ratio to match Parquet without altering the dense array structure.

## Spatial Chunking vs. Row-Level Random Access

While Zarr v3 and Parquet are established formats, emerging technologies like Lance [12] are actively addressing spatial data limitations. Lance is a modern columnar format written in Rust specifically designed for machine learning and multimodal data. It natively integrates with GeoArrow. Lance is optimised for huge-scale data storage and rapid random access. In recent benchmarks, Lance demonstrated significantly faster random access times than Parquet for fetching specific blobs of data (like imagery chunks). If a spatial API requires feeding randomly shuffled data into an AI model for inference, Lance provides an optimised solution.

The distinction between Lance and Zarr for polygon queries relies on the difference between Spatial Locality and Row-Level Retrieval. When querying a polygon, the requirement is a clustered geographic neighborhood. Zarr is optimised for this via 2D spatial tiles (e.g., 512x512 pixels). The Rust backend fetches the required 2D tile via a single HTTP Range request. Once decompressed in memory, all necessary pixels are spatially contiguous.

Lance and Parquet are 1D columnar formats. Querying a polygon requires translating the 2D shape into thousands of 1D Hilbert integers. Because the irregular polygon boundary cuts across the Hilbert curve, these integers are fragmented, requiring Lance to execute thousands of non-contiguous row fetches. The overhead of assembling these fragmented rows limits its performance for substantial batch polygon extractions (~0.46 seconds for 1K polygons, compared to Zarr's 0.24 seconds). Furthermore, because Lance prioritises raw memory layouts to guarantee microsecond seek times rather than using aggressive Dictionary and Run-Length Encodings like Parquet, its storage footprint for dense scalar grids is significantly larger (the uncompacted X/Y test dataset bloated to 4.7 GB in Lance compared to 528 MB in Parquet).

#### The Read Amplification Paradox (Overfetching)
A common critique of Zarr is that it suffers from "overfetching" (Read Amplification). If a polygon only touches 10 pixels within a 512x512 chunk, Zarr still must download and decompress the entire 262,144-pixel chunk. 

While this is true, it is a paradox that Parquet and Lance actually overfetch *more* data for 2D geometries. Columnar formats compress data into discrete blocks (e.g., 1MB Parquet Pages). To read a single row, the database must download and decompress the entire 1MB Page. Because a 2D polygon fragments into thousands of disconnected 1D Hilbert segments, those segments scatter across dozens of different compression pages on disk. 

To empirically prove this, I tracked the exact disk I/O required to extract a single polygon requiring just **23 KB** of raw pixel data (see Figure 3):

![Overfetch Comparison](/plots/06_overfetch_comparison.png)
*Figure 3: Log-scaled comparison of the uncompressed data fetched by Zarr (at various 2D chunk sizes) versus Parquet (1D Row Groups) to extract a 23 KB polygon.*

*   **Zarr (512x512 chunks):** Touched 1 chunk, downloading **0.50 MB** of uncompressed data.
*   **Parquet (Row Groups):** The 1D Hilbert line wandered so widely that the polygon's min/max bounds spanned 735 different Parquet Row Groups. DuckDB was forced to download and decompress **689 MB** of irrelevant data just to find the 23 KB it needed!

**The Parquet Zone Map Trap:** Why did DuckDB read 689 MB of data? Parquet uses "Zone Maps" (min/max metadata per Row Group) to quickly skip irrelevant data (Predicate Pushdown). When DuckDB queries the polygon, it looks at the absolute lowest and highest Hilbert integer inside the polygon and tells Parquet to fetch every Row Group that falls within that range. Because the 1D Hilbert curve snakes back and forth across the 2D map, a small geographic polygon can span a substantial numeric range. DuckDB fetched 735 Row Groups, decompressed them into RAM, and then realised 99% of the rows belonged to neighboring towns outside the polygon. 

**The Coordinate Tax Reconsidered (2D Zone Maps):** Is there a way to solve this Parquet overfetching without abandoning the format? Yes, but it requires paying the "Coordinate Tax" you might initially try to avoid. If we add the explicit X and Y coordinates (or a coarse 2D `grid_id`) back into the uncompacted `q2500` Hilbert Parquet file, the file size bloats from 319 MB to roughly 528 MB. However, because Parquet maintains Zone Maps for *every* column, you can structure your query to include a 2D bounding box: `WHERE x >= min_x AND x <= max_x AND z_index IN (...)`. DuckDB will evaluate the X/Y Zone Maps first, allowing it to instantly prune the hundreds of irrelevant Row Groups that the 1D Hilbert index dragged in. I empirically tested this, and DuckDB's scan time dropped to near zero. This represents the ultimate architectural tradeoff: you can pay for storage (528 MB for the uncompacted Parquet file with X/Y coordinates) to fix your I/O, or you can use Native Zarr to keep the file size small (1.7 MB for `q2500` Zarr) while inherently maintaining 2D chunk bounds.

Therefore, to read a single polygon, a columnar database relying solely on a 1D spatial index downloads and decompresses significantly more irrelevant data blocks than Zarr, which cleanly bounds the overfetching to a few 2D spatial chunks.

**Tuning the Spatial Chunk Size (Read vs. Request Amplification):** It is important to note that the 512x512 chunk size is simply a default designed to balance compression ratios with read speeds. If a production API primarily queries very small polygons, fetching a 0.50 MB chunk is still an overfetch. The logical solution is to configure the Zarr writer to tune the chunk size based on the typical polygon size. In my simulation for the exact same 23 KB polygon, configuring Zarr with `chunks=(128, 128)` caused it to touch 4 chunks, reducing the total download size to just **0.12 MB**. 

However, doing so introduces the fundamental trade-off of cloud I/O: **Read Amplification vs. Request Amplification**. 
While smaller chunks significantly reduce the number of wasted bytes downloaded, they increase the number of HTTP Range requests required to cover a polygon (e.g., 4 requests instead of 1). In cloud storage like S3, Time-To-First-Byte (TTFB) latency is exceptionally high. Making 100 small HTTP requests is almost always slower and more expensive than making 1 large HTTP request. 

Data engineers must tune the chunk size to find the exact sweet spot for their specific API payloads. Conversely, while Parquet's block sizes (Data Pages and Row Groups) can also be shrunk to reduce 1D overfetching, doing so breaks the contiguous lengths required for its columnar Run-Length Encoding (RLE) and dictionary compression, causing the file size to increase substantially. Zarr's 2D chunks avoid this tension, allowing for true geometric tuning.

**General Heuristic:** For fetching disconnected data points (e.g., AI model data shuffling), Lance is optimal. For fetching clustered 2D shapes, Zarr is highly efficient.

### Scaling Performance (Batch Extractions)

With the data stored as compacted 1D integers in Parquet, I implemented a client-side region coverer. The system calculates the exact Hilbert cells intersecting the requested polygons and passes that list to Polars or DuckDB to execute a vectorized Hash Join. By comparing this tabular approach against traditional geospatial masking and emerging formats, we can observe the fundamental scaling constraints of each architecture.

![Polygon Scaling](/plots/05_scaling_polygons.png)
*Figure 5: Scaling execution time as a function of polygon batch size. Database join logic scales sub-linearly, while traditional bounding box masks scale linearly.*

As seen in the scaling chart (see Figure 5), the performance divergence is stark:

1. **Traditional GIS (Rasterio + COG):** Scales strictly linearly ($O(N)$). Sequential bounding box masking struggles under the weight of large batches, requiring 0.73 seconds for 1,000 polygons (see Figure 5).
2. **Tabular Data (Polars/DuckDB + Parquet):** The database query step (the Hash Join) scales sub-linearly (empirically $O(N^{0.83})$). Polars processed the database query for 1,000 polygons in just 0.18 seconds (see Figure 5). This is primarily because 466 million raw DEM pixels were compacted down to roughly 2.6 million parent cells, mathematically shrinking the problem space before the join operation.
3. **AI Random Access (Lance):** Lance entirely bypasses the metadata startup penalty of Parquet. For small batches (10 polygons), its random-access design executes the query almost instantly. However, as the batch scales to 1,000 polygons, the overhead of assembling thousands of fragmented rows catches up, pulling its execution time to 0.25 seconds (slightly slower than Parquet) (see Figure 5).
4. **Native Chunking (Pure Rust Zarr):** Bypassing both SQL overhead and Python iteration, the native Rust Zarr extraction dominates the pack. It maintains a functionally flat scaling curve due to minimal constant-time overhead, completing the entire extraction for 1,000 polygons in 0.24 seconds (see Figure 5).

**Why This Matters (The Extrapolation):** At a glance, a 0.49-second difference between Rasterio (0.73s) and Zarr (0.24s) for 1,000 polygons might seem too trivial to justify an entirely new Rust-based architecture. However, this local benchmark masks a systemic failure at scale. Because Rasterio scales linearly ($O(N)$), extracting 10,000 polygons will take ~7.3 seconds, and 100,000 polygons will take ~73 seconds—completely exceeding synchronous API constraints. Conversely, Zarr's sub-linear efficiency easily absorbs substantial batches. Furthermore, Rasterio achieved 0.73 seconds on a local NVMe drive. Over a cloud network (like AWS S3), fetching data introduces Time-To-First-Byte (TTFB) latency (often ~50ms per request). Standard GIS tools querying Cloud Optimised GeoTIFFs (COGs) must download complex TIFF headers to understand the file layout before fetching the actual tiles. When processing 1,000 polygons across multiple files, Rasterio often executes these network requests sequentially, paying the 50ms TTFB penalty thousands of times. This causes cloud extraction times to increase substantially into minutes. The pure-Rust Zarr architecture is effectively immune to these cloud I/O penalties. Zarr separates its metadata from its data. The Rust backend downloads a single, instantly readable JSON metadata file. It then uses simple mathematics to calculate exactly which byte-ranges are needed for all 1,000 polygons. Because it is written in asynchronous Rust, it fires all 1,000 HTTP Range requests *concurrently*. It pays the 50ms network latency penalty exactly once, completely hiding the cloud I/O bottleneck.

## Scaling Back to the Globe

Returning to the original 240 TB multi-dimensional problem (over 700 distinct hazard layers per 30m spatial pixel), the architectural path requires evaluating the limits of modern data structures. Scaling the Hilbert Parquet architecture globally introduces a hyper-dimensionality challenge. Parquet schemas are designed for narrow tables. Storing over 700 layers per pixel requires either an exceptionally wide table (destroying columnar scan efficiency) or a deeply nested array structure (forcing the database to deserialize large blobs of data to read a single variable). Conversely, Zarr natively handles N-dimensional chunking (e.g., `[layer, y, x]`). If an API request specifies a specific layer, the Rust backend calculates the 1D index for that slice and reads only the corresponding bytes, leaving the other layers untouched.

### Overcoming Cloud Storage Limits: Zarr v3 Sharding
A final consideration for deploying a global Zarr cube is the nature of cloud object storage. 

Zarr v2 stores every chunk as an individual file. A global 30m dataset with over 700 layers chunked into 512x512 grids would generate billions of files. In a key-value object store like S3, this volume of files exhausts inode limits on caching layers, slows data transfer, and increases costs associated with HTTP `PUT` and `LIST` requests. The standard solution is Zarr v3 Sharding [11]. Sharding introduces a two-tier hierarchy by grouping thousands of chunks into larger "Shard" files (e.g., 500 MB). A byte-offset index is appended to the end of each shard. 

When the Rust backend queries the globe, it:
1. Reads the master metadata to locate the relevant Shard file.
2. Fetches the trailing byte-offset index via an HTTP Range request.
3. Parses the index to locate the byte-offset of the specific 512x512 inner chunk.
4. Executes a second HTTP Range request to fetch the required bytes.

This approach preserves the read performance of small chunks while significantly reducing the total number of files in the storage bucket. Furthermore, because global hazard data typically follows a "Write Once, Read Many" (WORM) lifecycle, the shard index never changes. The Rust backend can aggressively cache these indices in RAM. Because user queries are frequently localised to common geographic tiles or regions, this caching strategy is highly effective. It entirely neutralises the "double-fetch" penalty of Sharding, allowing all subsequent API queries within that shard to instantly locate the inner chunk and fetch the required data with a single HTTP Range request.

## Conclusion

Building a synchronous, high-concurrency API over petabytes of spatial data requires pushing modern data engineering to its absolute limits. As I have demonstrated, relying on industry-standard GIS raster pipelines (like Rasterio and COG) introduces sequential $O(N)$ geometry masking bottlenecks that fail under the weight of substantial polygon batches. Beyond compute bottlenecks, extracting COGs over cloud object storage like S3 forces the system to sequentially fetch complex TIFF headers and decode scattered tiles, compounding Time-To-First-Byte (TTFB) latencies until query times increase to minutes. Zarr fundamentally resolves this by separating its metadata into a single, instantly readable JSON file, allowing asynchronous backends to calculate byte-offsets and fire thousands of highly concurrent HTTP Range requests—effectively neutralising the cloud network penalty. 

Flattening spatial arrays into 1D columnar formats like Parquet and executing vectorized Hash Joins offers a promising sub-linear alternative. However, this approach forces engineers into difficult compromises. To achieve extreme compression, one must use Quadtree Compaction, which destroys the area-weighting accuracy required for rigorous `AVG()` analytics. Furthermore, mapping 2D geometries onto 1D space-filling curves triggers the "Zone Map Trap," causing severe Read Amplification as the database overfetches hundreds of megabytes of fragmented row groups. Fixing this requires paying the "Coordinate Tax"—bloating the file size with explicit X/Y columns to enable 2D Zone Map pruning.

Ultimately, the most elegant architecture respects the physical reality of the data. For large-scale, multi-dimensional spatial analytics, a pure-Rust backend querying sharded N-dimensional Zarr v3 arrays provides the optimal, robust solution. 

By aggressively tuning the spatial chunk size to match the API's expected payload, using Zarr v3 Sharding to bypass cloud object storage limits, and caching the WORM index in memory to neutralise read latency, data engineers can achieve a robust balance of storage compression, mathematical accuracy, and microsecond API performance.
